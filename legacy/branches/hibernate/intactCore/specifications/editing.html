IntAct project component specification: Editing tool

This specification will gradually be completed. It describes the desired features of the editing tool for IntActCore. The features should be tested with IntAct Test, then applied to the full model.

  1. Technical requirements

    1. The tool should be a web-based application, using Servlet technology.
    2. The system should use HTML 4.0.
    3. Javascript may be used only for optional features, the system should be functional with Javascript turned off.
    4. As a persistence mechanism, the IntAct API should be used, avoiding direct SQL table access.

  2. Data volume

    1. One submission can comprise up to 1000 objects, from all of the externally visible IntAct classes.

  3. Functionality

    1. One submission may be the result of a complex editing process, but should only be made persistent on activation of a final "submit" button.
    2. At the beginning of the editing process, the user should get a session ID on screen and by email. Alternatively, incomplete submissions could be saved attached to the submitter login, and the submitter could get a list of all his recent or incomplete submissions.
    3. At any point in the editing process, an interruption of the process, through server or client failure, or timeouts, should only loose the data collected since the last server interaction, and should usually not represent more than ca. 10 minutes of lost work. Planned suspension of the editing process, and restart much later, should also be available. Any interrupted or suspended session should be retrievable by the session ID (see use case 49.
    4. A graphical view of the submission, it's complete and incomplete components, is highly desirable.

    6. Retrieve existing objects by query, then include them into the current submission. Example: Use an existing protein object in a new interaction.
